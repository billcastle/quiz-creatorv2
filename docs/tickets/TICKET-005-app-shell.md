## Goal
Build the persistent **application shell** (ARCHITECTURE §7.2): a real `AppLayout` with a **collapsible left sidebar** (category section + nav links + minimize-to-icons) and a **top header** carrying the **theme switcher + custom-theme dropdown** (FR-39), and introduce the **`_app` / `_bare` pathless-layout split** so the shell wraps app pages while sign-in/sign-up and the quiz-answering view (built later) render chrome-free in a `BareLayout`. This turns the current minimal, root-level shell into the product's real navigation frame that every subsequent feature page (Explore, builder, profile, settings, admin) renders inside, and moves the theme switcher out of the design-system demo and into its permanent home in the header.

## Scope

**In scope:**

- **`_app` / `_bare` pathless-layout split (ARCHITECTURE §7.1 / §7.2).** Refactor the file-based route tree so there are two pathless layout routes:
  - **`_app`** — renders `AppLayout` (sidebar + header + `<Outlet/>`). The **existing routes move under it**: `/` (index) and `/design-system` become children of `_app` so they render with the shell. In Tanstack Router file-based routing this is a pathless layout route (an `_app.tsx` route file whose `component` is `AppLayout`, with child route files nested under it — e.g. `routes/_app/index.tsx`, `routes/_app/design-system.tsx`, OR the `_app.index.tsx` / `_app.design-system.tsx` flat-file naming — implementer picks whichever Tanstack file-naming convention the project's generator supports; **state which** in Implementation notes). Paths are unchanged (`/`, `/design-system`) — the `_app` segment is pathless.
  - **`_bare`** — renders a minimal `BareLayout` (no sidebar/header, just `<Outlet/>` centered/plain) for routes that must not show the shell: `/auth/sign-in`, `/auth/sign-up`, and the quiz-answering view (SPEC §7.2 / ARCHITECTURE §7.2). Since **those pages do not exist yet**, this ticket only creates the `_bare` pathless layout route + the `BareLayout` component (the pattern/slot). Add **at most one throwaway placeholder** child under `_bare` **only if** the router/generator requires a child for the layout route to register (e.g. a trivial `routes/_bare/placeholder.tsx` returning a stub) — prefer NOT adding one if the generator is happy with a childless layout route; **state the call**. Do **NOT** build auth or answering pages here.
  - **`__root.tsx`** becomes a **pass-through root** (renders just `<Outlet/>` and any app-global providers) instead of rendering `AppLayout` directly (it currently renders `AppLayout`). The layout choice moves down to `_app` / `_bare`. Keep the root minimal.

- **`AppLayout` shell (replaces the current minimal one).** Upgrade `frontend/src/components/layout/AppLayout.tsx` from the current `<main><Outlet/></main>` stub into a real layout: a **persistent left `Sidebar`** + a **top `Header`** + a scrollable **main content region** hosting `<Outlet/>`. Responsive: on small screens the sidebar may collapse to icons (reuse the same collapse mechanism) — full mobile drawer/`Sheet` behavior is a nice-to-have and MAY be deferred (see Technical notes; `Sheet` is a deferred batch-2 primitive — do not add it just for this).

- **`BareLayout` component.** A new `frontend/src/components/layout/BareLayout.tsx` — a minimal wrapper (`min-h-screen bg-background text-foreground`, centered content) rendering `<Outlet/>` with **no** sidebar/header. This is the chrome-free frame for auth + answering.

- **Collapsible `Sidebar` component (SPEC §7.2 / sidebar requirements).** A new `frontend/src/components/layout/Sidebar.tsx` (extract sub-pieces only if it genuinely helps — keep it lean) providing:
  - **(a) Nav links section** — links only to routes that EXIST today: **Home (`/`)** and **Design System (`/design-system`)**. Use Tanstack Router `<Link>` with active styling (`activeProps`/`data-status`). Do **NOT** add links to not-yet-built pages (Explore, builder, profile, settings, admin); if a placeholder for the eventual nav is wanted, it may be a **disabled/commented stub with a TODO** — do not link to non-existent routes (would break navigation).   - **A custom-theme dropdown** that lets the user choose among the themes registered in `frontend/src/themes/manifest.ts` (currently `light`, `dark`) — built so adding a theme file + manifest entry makes it appear automatically (drive the dropdown from `THEMES`, don't hard-code entries). **Recommendation: build the dropdown with the existing shadcn `Select`** (already present from TICKET-004) to avoid pulling in `DropdownMenu` (a deferred batch-2 primitive). It is acceptable for the toggle to be a fast light↔dark switch and the dropdown to be the full "choose a theme" control. State the interaction model chosen.

If you extend `applyTheme`/`getStoredTheme` this way:
  - Keep the **anti-FOUC inline `<script>`** in `frontend/index.html` in sync (it must apply BOTH the `.dark` class and the `data-theme` attribute pre-paint from the persisted `qc-theme` value, still `try/catch`-guarded and dependency-free).

Only light+dark ship in this ticket — do NOT author new theme `.css` files here (adding a third theme is its own trivial follow-up); just make the machinery + dropdown ready for them.
  - This is a **surgical generalization**, not a rewrite: if light/dark is all that's wired and the `[data-theme]` path is stubbed/ready-but-unused, that is acceptable **as long as** the dropdown enumerates `THEMES` and switching among the shipped themes works. State exactly what was generalized vs stubbed.

## Technical notes (architect)
- **Alignment:** ARCHITECTURE **§7.1** (route tree + `_app`/`_bare` layout groups — this ticket lands the split, moving `/` and `/design-system` under `_app` and creating the `_bare` slot), **§7.2** (AppLayout vs BareLayout; collapsible sidebar hidden on auth + answering; header with theme switch + custom-theme dropdown; sidebar collapsed state in Zustand + localStorage), **§7.3** (Zustand `uiStore` for sidebar/modals; `themeStore` optional — kept in `lib/theme.ts` for now; Tanstack Query is server-state only and is NOT used here), **§8.2** (folder tree: `components/layout/`, `stores/`). **ADR-010** (theming: build-time token files + runtime switch toggling a `data-theme`/class on `<html>`; manifest-registered themes). **SPEC FR-39** (theme switch + custom-theme dropdown top-right), **FR-37/38** (token themes, light default + dark), **NFR-5** (a11y — keyboard-operable nav, labels/`aria` on the collapse toggle and icon-only items), **NFR-6** (responsive).
- **File-based layout split — how (Tanstack Router):** the current `__root.tsx` renders `AppLayout` as the root `component`; that is why every route gets the shell. To split: make `__root` a **pass-through** (`component` renders `<Outlet/>` only), then add a **pathless layout route `_app`** whose `component` is `AppLayout`, and nest the existing routes beneath it; add a sibling **pathless layout route `_bare`** whose `component` is `BareLayout` for future auth/answering routes. Use the file-naming convention the project's generator (`@tanstack/router-plugin`) already emits — either the **directory form** (`routes/_app/index.tsx`, `routes/_app/design-system.tsx`, `routes/_bare/…`) or the **flat dotted form** (`routes/_app.index.tsx`, `routes/_app.design-system.tsx`). Pick ONE consistently; `routeTree.gen.ts` will regenerate. Verify the generated tree still resolves `/` and `/design-system` at the same paths (the `_app` segment is pathless → URLs unchanged). This is the **breaking-structure** part of the ticket — do it first and confirm the app still renders before layering in the sidebar/header.
- **Sidebar: shadcn `Sidebar` vs custom — RECOMMENDATION: build a lean CUSTOM sidebar** from existing primitives + Tailwind tokens + `lucide-react`, NOT the shadcn `sidebar` block. Rationale: shadcn's `sidebar` pulls a **large dependency surface** (its own provider/context, `Sheet` for mobile, `Tooltip`, `Skeleton`, keyboard-shortcut plumbing, cookie-based state) — most of which is deferred batch-2 or duplicates state we're putting in Zustand. A custom sidebar (a `<nav>` with token-driven `bg-sidebar`/`bg-card`, `Button`/`Separator` we already have, `lucide` icons, width toggled by `sidebarCollapsed`) is **simpler, matches the design-system tokens directly, and keeps the dependency/graph small** — consistent with CLAUDE.md's simplicity guardrails. If the implementer finds the custom collapse ergonomics costly, the fallback is the shadcn sidebar, but it must NOT drag in unrelated batch-2 primitives as a side effect; **state the choice + rationale**.
- **Theme dropdown: `Select` vs `DropdownMenu` — RECOMMENDATION: `Select`.** `Select` already exists (TICKET-004), is form-semantic, and enumerating `THEMES` in it needs zero new deps. `DropdownMenu` is a deferred batch-2 primitive; adding it here (only for this) is acceptable but unnecessary. Prefer `Select`; if `DropdownMenu` is used, pin its `@radix-ui/react-dropdown-menu` **exact** and note it lands early by exception.
- **Extending `lib/theme.ts` for named themes + anti-FOUC sync:** ADR-010 says the switch toggles "a `data-theme`/class on `<html>`". Today `lib/theme.ts` only toggles `.dark`. Generalize so: **`applyTheme(theme)`** applies `.dark` for the dark built-in **and** sets `document.documentElement.setAttribute('data-theme', theme)` (or removes it for the default) for any named theme, persisting to `qc-theme`; **`getStoredTheme()`** validates against `THEMES` keys (not a hard-coded `"light"|"dark"`). **Critically, the inline anti-FOUC `<script>` in `index.html` must mirror this exactly** — it must read `qc-theme` and set BOTH the `.dark` class and the `data-theme` attribute pre-paint so no flash occurs when a non-default theme is later added; keep it tiny, `try/catch`-guarded, dependency-free, and using the same `qc-theme` key. **Keep `lib/theme.ts` and the inline script in lockstep** (same key + same mechanism) — drift here reintroduces the FOUC nit. Do not touch `index.css`'s `@custom-variant dark` / `@theme inline` ordering (TICKET-003 nit #3) unless strictly necessary.
- **Zustand + persist:** `create<UiState>()(persist(..., { name: "qc-ui" }))`. Keep the persisted shape minimal (`{ sidebarCollapsed }`) and versioned-safe (a single boolean). Use a distinct localStorage key from `qc-theme` (e.g. `qc-ui`) so theme and UI persistence stay independent. Pin `zustand` exact. No selectors library, no immer, no devtools middleware — keep it minimal.
- **`@/*` imports + conventions:** all new files use the `@/*` alias (agrees across Vite/vitest/tsconfig/components.json per TICKET-003). Strict TS: `import type` for type-only imports (`verbatimModuleSyntax`), satisfy `noUncheckedIndexedAccess` (e.g. array access in the category/theme maps) and `exactOptionalPropertyTypes`. Biome-clean; no blanket ignores; `routeTree.gen.ts` stays in the existing Biome ignore.
- **a11y (NFR-5):** the collapse toggle needs an `aria-label` and (ideally) `aria-expanded`; icon-only items need `title`/`aria-label` so the label is discoverable when collapsed; nav uses a `<nav>` landmark; active link exposes `aria-current="page"` (Tanstack Router `activeProps` can set it). The theme controls need accessible labels. Keep it basic but correct.
- **Carried follow-up (do NOT resolve here):** TICKET-004 nit (c) — the **canonical form-field label pattern** (nesting vs `htmlFor`+`id`) — belongs to the **builder/forms ticket**, not this shell ticket. Note it, don't decide it. This ticket adds no forms.
- **Simplicity guardrails (CLAUDE.md):** shell + split + one sidebar + one header + one ui store + theme wiring only. No pre-created empty route files for future pages, no account-menu logic, no Query hooks, no `lib/api.ts`, no third theme file, no `Sheet`/mobile drawer, no batch-2 primitives beyond (at most) the `DropdownMenu` alternative. Every changed line traces to "the app now has a persistent shell with a working sidebar collapse + header theme switcher, and the layout split is in place."
- **Recommended split if the session overruns:** land the **`_app`/`_bare` split + `AppLayout`/`BareLayout` + collapsible `Sidebar` + `ui` store + tests** in THIS ticket, and fast-follow the **header custom-theme dropdown + `lib/theme.ts` named-theme generalization** as a small immediate follow-up (keeping just a light/dark toggle in the header here). Attempt the full scope first; **record any deferral with rationale**.
- **Risks to watch (flag in Implementation notes):**
  1. **Route-tree refactor is the risky part** — moving `/` and `/design-system` under `_app` regenerates `routeTree.gen.ts`; verify URLs are unchanged and both routes still render before adding shell UI. If the generator's pathless-layout naming trips up, resolve via the correct file-naming convention, not by abandoning the split.
  2. **Anti-FOUC ↔ `lib/theme.ts` drift** — if you generalize to `[data-theme]`, the inline script MUST mirror it; a mismatch reintroduces the flash. Keep them in lockstep.
